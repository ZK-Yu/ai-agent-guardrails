# CLAUDE.md

用于减少 LLM 常见编码失误的行为准则。可按需与项目特定说明合并使用。

**权衡：** 这些准则更偏向谨慎而不是速度。对于琐碎任务，请自行判断。

## 1. 编码前先思考

**不要想当然。不要掩饰困惑。把权衡讲清楚。**

在开始实现之前：
- 明确写出你的假设。如果不确定，就提问。
- 如果存在多种解释，先列出来，不要默默选一种。
- 如果有更简单的方法，直接说明。必要时提出异议。
- 如果有不清楚的地方，就停下来。指出困惑点并提问。

## 2. 简单优先

**只写解决问题所需的最少代码。不要做任何猜测性的扩展。**

- 不要添加需求之外的功能。
- 不要为只使用一次的代码做抽象。
- 不要加入未被要求的“灵活性”或“可配置性”。
- 不要为不可能发生的场景加错误处理。
- 如果你写了 200 行，但 50 行就能解决，那就重写。

问自己一句：“一个资深工程师会觉得这写得太复杂吗？” 如果答案是会，那就继续简化。

## 3. 外科手术式改动

**只改必须改的地方。只清理你自己造成的问题。**

在编辑现有代码时：
- 不要顺手“改进”相邻代码、注释或格式。
- 不要重构那些本来没坏的东西。
- 遵循现有风格，即使你个人会写得不同。
- 如果你注意到无关的死代码，可以提一句，但不要删除。

当你的改动产生遗留项时：
- 移除因为你的改动而变成未使用的 import、变量或函数。
- 不要删除原本就存在的死代码，除非用户要求。

判断标准：每一行变更都应该能直接追溯到用户请求。

## 4. 以目标驱动执行

**先定义成功标准，再循环验证直到达成。**

把任务转成可验证的目标：
- “加校验” -> “先为非法输入写测试，再让测试通过”
- “修 bug” -> “先写一个能复现问题的测试，再让它通过”
- “重构 X” -> “确保改动前后测试都通过”

对于多步骤任务，先写一个简短计划：
```text
1. [步骤] -> 验证：[检查项]
2. [步骤] -> 验证：[检查项]
3. [步骤] -> 验证：[检查项]
```

强成功标准能让你独立推进。弱成功标准（比如“把它弄好”）则会不断需要澄清。

## 5. 子目录说明文档

**创建子目录时，先创建该目录自己的 `CLAUDE.md`。**

如果创建了子目录（如 `Util`、`Constant` 或 `Database` 等），需要先在该目录内创建一份 `CLAUDE.md` 文档。

如果是在处理一个已经存在的项目，发现某个子目录还没有 `CLAUDE.md`，也需要先补上一份，再继续修改该目录中的代码。

初始化该目录的 `CLAUDE.md` 时，文档内部需要包含以下内容：
- 如果修改此目录里面的代码文件，那么请同步更新此 MD 文档。
- 此目录的代码简介，例如该目录中存放了实现哪些功能的代码，内容要简洁。
- 此目录的结构树。

## 6. Python 环境与依赖

**始终使用当前目录的 `.venv`，并保持依赖声明同步。**

- 处理 Python 项目时，一定要使用当前目录下的 `.venv` 虚拟环境。
- 如果当前目录还没有 `.venv`，先创建虚拟环境并激活，再进行安装依赖、运行脚本、执行测试等操作。
- 安装任何依赖后，一定要同步更新 `requirements.txt`。
- 如果调整了依赖版本或删除了依赖，也要同步维护 `requirements.txt`，确保环境声明与实际安装状态一致。

---

**这些准则生效的表现是：** diff 里不必要的改动更少，因过度复杂而返工的情况更少，而且澄清问题会出现在实现之前，而不是出错之后。
